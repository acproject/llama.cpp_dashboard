# 当前能力与可用场景说明

## 1. 一句话概括

当前这个系统已经不只是一个 llama.cpp 面板，而是一个围绕 **模型服务管理、OpenAI 兼容网关、透明 RAG 注入、Agent 注册与编排、任务运行态追踪、任务队列执行、Nginx 入口管理** 逐步成型的轻量控制台。

如果用更直白的话说，它现在已经可以拿来做：

- 管理一组本地或局域网内的 llama.cpp 服务
- 通过统一入口对外提供 OpenAI 兼容 API
- 通过透明 RAG 网关让普通模型请求自动获得知识库上下文
- 用 Agent 的方式组织“谁来处理什么请求”
- 把一次请求完整记录成 Run，并追踪它的事件时间线
- 把请求进一步落成 Task，支持排队、认领、完成、失败、结果记录
- 在监控界面里同时观察服务、Run、Task 的运行状态
- 生成和维护 Nginx 反向代理配置
- 给 Agent 录入和管理来自开源项目的能力来源

---

## 2. 当前已经具备的核心能力

### 2.1 llama.cpp 服务注册与管理

系统可以把多个 llama.cpp 实例统一注册起来，并管理它们的基础信息与能力标签，例如：

- 服务名称、主机、端口、模型
- 服务状态
- 权重
- 能力标签
- 健康检查结果

这意味着你可以把原本零散的多个模型实例，收拢成一个统一可管理的服务池。

### 2.2 OpenAI 兼容网关

系统已经具备 OpenAI 风格的兼容入口，可以把外部请求统一接入，然后由内部调度逻辑决定把请求交给哪个 Agent、哪个 Service 处理。

它的价值在于：

- 上游应用不需要直接感知底层到底有几个 llama.cpp 实例
- 你可以在不改调用方代码的前提下调整路由策略
- 请求过程可以被记录成结构化运行态数据

### 2.3 Agent Registry

系统现在已经支持建立 AgentProfile，用来定义：

- Agent 名称
- 角色
- 描述
- 系统提示
- 默认模型
- 绑定服务
- 能力标签
- 工具标签
- 是否启用

这让系统不再只是“挑一个服务”，而是开始具备“先选一个合适的 Agent，再按 Agent 约束去选服务”的能力。

### 2.4 RAG 管理与透明注入

系统现在已经具备独立的 RAG 管理能力，并且已经把它接到了 OpenAI 兼容网关里。

当前已经支持：

- 创建和管理 RAG collection
- 为 collection 选择已注册的 embeddings 服务
- 导入文档、切块、向量化并写入 MiniMemory
- 利用图节点与标签做增强检索
- 在请求中携带 `rag.collectionId`
- 网关自动先检索上下文
- 将 `contextText` 自动注入 chat/completion prompt
- 在运行态里记录 RAG 集合与命中摘要

这意味着系统已经不只是“有一个知识库”，而是已经开始具备“对上游模型透明地增强回答上下文”的能力。

### 2.5 运行态追踪

每次请求已经可以形成 Run 记录，并查看：

- Run 主记录
- 状态变化
- 时间线事件
- 关联的 Service
- 关联的 Agent
- 关联的 Task

这非常适合排查“某次请求为什么失败”“为什么命中了这个模型”“为什么发生重试”这类问题。

### 2.6 Task Dispatcher / 任务执行闭环

系统已经具备轻量任务编排能力，支持：

- 创建任务主记录
- 写入任务事件流
- 建立父子任务
- 入队
- 显式认领
- 按队列 claim next
- 写入租约
- 写入执行结果
- 标记完成或失败

这意味着它已经不仅是请求代理，也开始具备“任务控制面”的雏形。

### 2.7 统一监控面板

监控页当前已经能同时看到多类运行信息：

- 服务状态
- Run 列表
- Task 列表
- 队列统计
- 执行器操作
- 任务详情
- 任务与 Run 的双向关联

因此它很适合作为本地 Agent 系统的“控制塔”。

### 2.8 Nginx 配置管理

系统已经可以生成和管理 Nginx 配置，用于把服务池统一暴露成一个入口。

适合的场景包括：

- 给多个模型实例统一挂一个反向代理入口
- 让上层应用只面向一个网关地址接入
- 结合服务状态动态维护 upstream 配置

### 2.9 开源能力来源导入与管理

系统现在已经支持从以下两个资料源导入能力来源到 Agent：

- `docs/agency-agents`
- `docs/CLI-Anything`

当前已经可以做到：

- 浏览可导入条目
- 搜索和筛选来源
- 把来源导入到 Agent 表单
- 自动合并一部分能力标签与工具标签
- 在 Agent 卡片里查看已导入来源

这一点很重要，因为它让系统具备了“把公开能力资产整理进自家 Agent Registry”的基础。

### 2.10 关键能力矩阵

围绕当前系统最核心的 5 项底层能力，可以先做一个比较明确的实现判断，并顺手标出下一阶段最自然的演进方向：

| 能力 | 当前状态 | 当前已经做到的部分 | 当前边界 | 下一步最自然演进方向 |
| --- | --- | --- | --- | --- |
| 任务状态机 | 已实现 | 已支持 `pending / queued / running / completed / failed / cancelled`，并且能写入事件、租约、结果与完成状态 | 仍属于轻量任务编排，不是复杂工作流引擎 | 增加更细的中间态、重试策略、超时策略和状态转换约束，减少执行链路里的隐式状态 |
| 任务依赖 DAG | 部分实现 | 已支持 `dependsOnTaskIds`、父子任务关系、claim 前依赖校验 | 还没有完整的 DAG 调度器、拓扑编排、失败传播策略与依赖图可视化 | 补齐依赖图调度、失败传播、批量释放下游任务和 DAG 可视化，让它真正进入工作流编排层 |
| lease / queue | 已实现 | 已支持入队、显式 claim、按队列 `claim-next`、租约写入、租约释放 | 目前主要通过 API 模拟执行器，不是独立 Worker 集群 | 引入独立 Worker / Executor、心跳续租、批量 claim 和更原子的执行器动作接口 |
| 共享记忆 | 部分实现 | MiniMemory 已提供 KV、Meta、Tag、Graph、Embedding/Evidence Search 等底座能力 | 还没有统一的共享记忆模型、专门的 memory API 和协作记忆 UI | 定义统一 memory schema，补 run summary / fact / artifact / evidence 的读写接口，并增加可检索的共享记忆视图 |
| 证据检索 | 已实现 | 已支持文档切块、向量化、`EVIDENCE.SEARCHF`、标签过滤、图节点增强召回，以及透明 RAG 注入 | 当前主要围绕 RAG collection 使用，还不是独立的通用证据中心 | 从“RAG 页面能力”扩展到“通用证据层”，支持更多入口复用、检索评估、召回调优和证据追踪 |

如果换成一句更务实的话来描述，就是：

> 系统已经具备任务状态机、队列与租约、RAG 证据检索等可运行能力；任务依赖 DAG 和共享记忆也已经有基础结构，但还处在“轻量实现 / 基础设施已到位、产品化未完全补齐”的阶段。

再进一步按优先级来排，我会建议这样推进：

- **近期最值得做**：优先补齐独立 Worker / Executor、心跳续租、任务失败传播和更清晰的状态转换约束，先把任务执行闭环做稳
- **中期最值得做**：补统一 memory schema、共享记忆 API、DAG 可视化和证据检索评估，让系统从“能跑”升级为“能协作、能观测、能调优”
- **最终形态方向**：把任务编排、共享记忆、证据检索和 Agent 路由真正收敛成一个可继承、可搜索、可复盘的多 Agent 协作中枢

## 3. 现在可以拿它做什么

下面是一些更接近实际使用的场景。

### 场景 A：把多台 llama.cpp 机器统一成一个可管理网关

你有几台机器分别跑着不同模型，过去需要记很多地址和端口。现在可以：

1. 把它们都注册到服务池
2. 给每个服务标能力标签
3. 通过统一网关暴露 OpenAI 兼容接口
4. 通过监控页看每个服务的状态

**你得到的结果**：

- 调用方只需要接一个入口
- 服务更容易替换和扩容
- 故障排查更直观

### 场景 B：把知识库透明接到统一网关

你可以：

1. 创建一个或多个 RAG collection
2. 给 collection 导入文档并建立向量索引
3. 让上游请求在 body 里附带 `rag.collectionId`
4. 由网关自动检索并把上下文注入 prompt

**你得到的结果**：

- 调用方不需要自己实现检索编排
- 同一个模型接口可以直接变成“带知识库”的接口
- RAG 是否生效、命中了多少片段，都可以在运行态里追踪

### 场景 C：按 Agent 来组织不同类型请求

例如你可以定义：

- 一个 `planner-agent` 负责规划
- 一个 `coder-agent` 负责代码生成
- 一个 `reviewer-agent` 负责审查

每个 Agent 可以绑定不同的服务、模型偏好、能力标签。这样请求进来后，不再只是“随便找个模型”，而是“先找适合的角色，再走到适合的模型”。

**你得到的结果**：

- 请求分工更清晰
- 角色边界更明确
- 后续更容易演进成多 Agent 协作

### 场景 D：把一次失败请求追到根因

当某个请求异常时，你现在可以：

1. 在 Run 页面找到这次请求
2. 查看事件时间线
3. 看它路由到了哪个 Agent / Service
4. 看它是否触发了 Task
5. 继续跳到关联 Task 查看租约、结果、错误

**你得到的结果**：

- 可以做结构化排障
- 可以知道问题出在路由、执行、RAG 注入还是结果写回

### 场景 E：把系统当成一个轻量任务控制台

如果你未来有外部执行器、脚本 Worker、Agent Worker，这个系统现在已经能先承担任务控制面的职责，例如：

- 某类任务进入指定队列
- 执行器按队列 claim next
- 执行完成后回写结果
- 失败时记录事件和错误
- 在监控页统一看队列与任务状态

**你得到的结果**：

- 已经可以支撑小规模任务编排实验
- 非常适合做 Agent Worker 原型验证

### 场景 F：把开源 Agent 能力资产沉淀进自己的注册表

你现在已经可以把公开的开源能力来源整理到 Agent Registry 里，例如：

- 从 `agency-agents` 导入某类 Agent Persona
- 从 `CLI-Anything` 导入某类 CLI 集成能力定义

导入后，这些来源会成为 Agent 的“能力背景”和“工具线索”，便于你统一管理团队内部到底有哪些能力资产。

**你得到的结果**：

- 不再靠手工记忆某个能力来自哪个开源项目
- Agent 能力来源更可追踪
- 后续真正接入执行器时更容易落地

### 场景 G：作为本地 AI 平台的运维面板

如果你的目标不是只跑一个 demo，而是维护一套长期可用的本地 AI 服务，这个系统已经可以承担：

- 服务台账
- 网关入口
- RAG 知识库入口
- 运行监控
- 请求追踪
- 任务编排
- 反向代理配置管理

**你得到的结果**：

- 从“若干脚本 + 若干端口”升级为“可运营的本地平台”

## 4. 几个具体例子

### 例子 1：做一个团队内部代码助手入口

你可以这样配置：

- 注册多个代码模型服务
- 新建 `coder-agent`
- 给它绑定代码相关服务
- 给它配置 `code`、`review`、`tools` 等能力标签
- 通过 OpenAI 兼容接口给团队 IDE 或脚本调用

这样一来，团队内部就有了一个统一的代码助手入口，而且你可以知道每一次请求到底是如何路由和执行的。

### 例子 2：做一个“任务可见”的推理网关

假设你希望一次复杂请求不只是被简单转发，而是形成任务记录，那么当前系统已经适合用来：

- 把请求映射成根任务
- 记录执行过程中的事件
- 为某次服务尝试生成子任务
- 跟踪完成状态和失败原因

这对构建“可解释的 AI 请求执行链”非常有帮助。

### 例子 3：做一个“模型透明可用 RAG”的知识助手入口

你可以这样配置：

- 先创建一个产品文档或项目知识库 collection
- 在聊天页或 API 请求里选择这个 collection
- 继续使用原来的 OpenAI 风格 chat/completions 调用方式
- 让网关自动完成检索与上下文注入

这样一来，前端聊天页和 API 调用方都不需要自己额外拼接知识库 prompt，就能直接获得带上下文的回答能力。

### 例子 4：做一个多 Agent 实验台

你可以先在 Registry 里维护：

- planner
- coder
- reviewer
- researcher

然后为不同 Agent 设置：

- 不同系统提示
- 不同默认模型
- 不同能力标签
- 不同服务绑定

这时虽然系统还不是一个完整自动协作框架，但已经是一个非常好的多 Agent 运行台和配置中心。

### 例子 5：做开源能力目录中心

如果你最近在看很多开源 Agent 项目、CLI 项目，过去可能会散落在 README、笔记和浏览器收藏夹里。现在可以：

- 把可复用能力来源登记到 Agent
- 记录这些能力来自哪里
- 在卡片里持续查看和管理

这样做的价值，不是立刻“自动执行所有外部能力”，而是先把能力资产沉淀为结构化管理对象。

## 5. 当前最适合的使用定位

如果要给它一个现实定位，我会把它定义成：

> 一个面向本地大模型服务、透明 RAG 网关与多 Agent 演进过程的轻量控制面。

它现在最适合的三类工作是：

### 5.1 本地模型平台化

适合把多个 llama.cpp 实例组织成一个统一入口和管理后台。

### 5.2 透明 RAG 接入与知识增强

适合把知识库能力挂到统一网关和聊天界面，让上游调用方用几乎不变的接口享受检索增强。

### 5.3 Agent 运行与编排实验

适合做小规模 Agent Registry、路由策略、任务队列和执行器模型实验。

### 5.4 可观测性与运维

适合追踪请求、任务、服务状态，并把排障过程可视化。

## 6. 需要明确的边界

为了避免高估当前能力，也需要明确几个边界：

- 目前已经能导入和管理开源能力来源，但这不等于所有外部开源项目都已经被自动执行接入
- 目前任务系统已经有队列、租约、结果和认领机制，但还更偏向轻量控制面，而不是完整分布式调度平台
- 目前已经支持 Agent 维度的组织与选路，但离“全自动复杂多 Agent 协作系统”还有演进空间
- 目前 RAG 已经能自动检索并注入上下文，但仍然属于单次请求级增强，还没有扩展到更复杂的多阶段检索与工具联动

换句话说，当前最强的地方是：

- **控制面**
- **注册面**
- **运行态可观测**
- **任务编排基础设施**

而不是“所有能力都已全自动执行”。

## 7. 结论

如果只用一句话总结现在“它能干什么”：

> 它已经可以作为一个本地大模型、透明 RAG 与 Agent 系统的统一控制台，负责服务接入、知识库增强、请求路由、运行追踪、任务编排、监控观测以及能力资产管理。

对于下一阶段，最自然的演进方向会是：

- 真正把更多外部能力接到执行链路
- 继续把透明 RAG 扩展到更多前端入口和运行链路
- 引入独立 Worker / Executor
- 强化多 Agent 协作链路
- 让开源能力从“已登记”进一步变成“可调用”

---

## 8. 基于当前能力，后续可以继续做什么

如果不是只把它当成一个演示面板，而是想顺着现在这套能力继续往前推，后续最值得做的方向大致有下面几类。

### 8.1 从“任务控制面”走向“真正可执行的任务系统”

当前已经有：

- Task 状态机
- 队列
- 租约
- claim next
- 结果写回

所以后续最自然的演进是：

- 增加独立 Worker / Executor 进程
- 增加任务心跳与自动续租
- 增加超时、重试、失败传播与回滚策略
- 让任务不只是“可见”，而是真正“可稳定执行”

这一步完成之后，系统会从“任务运行态控制台”更接近“轻量任务调度平台”。

### 8.2 从“轻量依赖关系”走向“可视化 DAG 编排”

当前已经支持：

- 父子任务
- `dependsOnTaskIds`
- 依赖满足后 claim

所以后续可以继续做：

- DAG 拓扑调度
- 节点级失败传播
- 下游批量解锁
- 依赖图可视化
- 工作流模板化

这样它就不只是“任务列表”，而会变成真正的工作流编排入口。

### 8.3 从“RAG 页面”走向“通用证据中心”

当前已经有：

- 文档切块
- 向量化
- 标签过滤
- 图节点增强召回
- 透明 RAG 注入

所以后续非常适合继续做：

- 独立的证据检索 API 与复用入口
- 检索评估与召回质量对比
- 证据命中追踪与引用链展示
- 多 collection 联合检索
- 面向任务和 Agent 的证据挂接

这一步会让系统从“有 RAG”升级成“有证据层”。

### 8.4 从“底层存储能力”走向“共享记忆中心”

当前已经有 MiniMemory 的：

- KV
- Meta
- Tag
- Graph
- Embedding / Evidence Search

所以后续可以进一步抽象成统一记忆层，例如：

- `run summary`
- `fact`
- `artifact`
- `evidence`
- `review comment`

再往前一步，可以补：

- 统一 memory schema
- 面向 Agent 的 memory API
- 共享记忆检索页
- 记忆回放与复盘视图

这会让系统从“单次请求增强”逐步走向“多 Agent 可继承上下文”。

### 8.5 从“Agent 注册中心”走向“多 Agent 协作台”

当前已经有：

- Agent Registry
- Agent 选路
- 服务绑定
- 能力标签
- 任务运行态

所以下一步可以继续做：

- planner / coder / reviewer / researcher 角色分工
- 多 Agent 任务拆分与回收
- Agent 间共享证据与共享记忆
- Agent 协作链路可视化
- 不同 Agent 的成功率、耗时和命中偏好分析

这会把系统从“能管理 Agent”推进到“能组织 Agent 协作”。

### 8.6 从“能力登记”走向“能力执行”

当前已经能把开源能力来源沉淀到 Agent Registry，但还主要停留在“登记、查看、管理”。

所以后续可以继续打通：

- 工具定义到执行器映射
- 开源能力到本地 Worker 的调用桥接
- CLI / HTTP / 脚本类工具统一接入
- 能力调用日志、权限控制与结果回写

这样这些能力来源就不再只是“参考资料”，而会真正进入执行链路。

### 8.7 从“本地面板”走向“本地 AI 中控平台”

把上面几层串起来之后，系统后续可以落到更完整的使用形态，例如：

- 团队内部统一模型入口
- 带透明 RAG 的知识助手平台
- 多 Agent 开发与评审实验台
- 带队列与执行器的任务编排中心
- 本地 AI 服务运维与观测平台

如果用一句话概括后续方向，可以把它理解成：

> 从“服务管理 + 网关 + RAG + 任务面板”，逐步演进成“带证据层、共享记忆和执行链路的本地 AI 协作中控平台”。

### 8.8 更偏执行的优先级路线

如果把上面的方向进一步压成更适合落地推进的版本，我会建议按下面三层来排：

#### P0：马上做

先优先做那些能明显提升系统稳定性和执行闭环完整度的事情：

- 引入独立 Worker / Executor，跑通任务真正执行链路
- 增加任务心跳、自动续租、超时回收
- 补齐失败传播、重试策略、状态转换约束
- 给任务执行链补更清晰的观测与错误定位信息

这一层的目标不是“功能变多”，而是先把任务系统从“能展示”做成“能稳定跑”。

#### P1：下一阶段做

当执行闭环稳定之后，再补真正拉开系统差异化的能力层：

- 补统一 memory schema 和共享记忆 API
- 把证据检索从 RAG 页面能力扩展成通用证据层
- 增加 DAG 可视化、依赖解锁和工作流模板化
- 增加 Agent 间共享证据、共享记忆和协作链路展示

这一层的目标是让系统从“单点能力集合”变成“可以协同工作的平台”。

#### P2：中长期做

最后再把前面的能力真正收敛成平台级形态：

- 打通开源能力来源到真实执行器的调用桥接
- 形成多 Agent 协作编排、执行、复盘一体化链路
- 建立统一证据中心、共享记忆中心和任务编排中心
- 让系统成为团队内部统一的本地 AI 中控平台

这一层的目标是从“本地控制台”升级到“本地 AI 协作基础设施”。
